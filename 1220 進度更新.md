## 🆕 3. 整合 Google Cluster Trace 真實資料集

### 為什麼需要 Google Cluster Trace？

PlanetLab 非常適合做學術基準實驗，但仍有以下限制：

- 僅提供 CPU 使用率，**沒有真實 PM 容量**
- VM / PM 規模偏小（~1000 VM）
- 無法完整呈現真實大型資料中心的異質性

因此本專案額外整合 **Google Cluster Trace (2011)**，用來：

- 補足 **真實資料中心規模**
- 使用 **真實 PM CPU / Memory 容量**
- 驗證演算法在 **工業級工作負載** 下的表現
- 強化專案的實務與論文等級完整性

---

### 資料集來源

- 官方資料集：Google Cluster Trace (2011)
- 廣泛用於 VM placement / scheduling / energy-aware cloud 研究
- 原始系統：Google Borg Scheduler

本研究使用以下檔案：

- `task_usage/part-00000-of-00500.csv.gz`
- `machine_events/machine_events.csv.gz`

（僅取子集以控制資料規模）

---

### Google Trace 中 VM 與 PM 的定義

#### VM 定義方式

在 Google Cluster Trace 中：

- 一個 **job (job_id)** 代表一個應用
- 每個 job 底下包含多個 **task (task_index)**

本研究定義：
(job_id, task_index) → 一個 VM

每個 VM 對應一個獨立的 placement 單位。

---

#### VM CPU / Memory 計算方式

**資料來源：** `task_usage.csv`

使用欄位如下：

```python
usecols = [0,1,2,3,4,5,6,7,10]
names = [
  "start_time",
  "end_time",
  "job_id",
  "task_index",
  "machine_id",
  "cpu_rate",
  "canonical_memory",
  "assigned_memory",
  "max_memory_usage"
]
●CPU 需求（VM CPU demand）：
cpu_demand = max(cpu_rate) × 100
●Memory 需求（VM memory demand）：
memory_demand = max(max_memory_usage) × 100
📌 備註：
canonical_memory 為請求值（request），非實際使用量，因此未採用。

PM（Physical Machine）容量計算方式

資料來源： machine_events.csv

Google Trace 中的 PM 容量為標準化比例值，本研究將其轉換為統一資源尺度：

●PM CPU capacity：
cpu_capacity = raw_cpu_capacity × 100
●PM memory capacity：
memory_capacity = raw_memory_capacity × 100
此轉換方式使 VM 與 PM 的資源單位一致，可直接用於 VM placement 演算法。

為什麼 Google Trace 也需要取樣？
Google Cluster Trace 原始規模極大（數十萬 VM / PM），直接使用會導致：

●FFD / BFD 執行時間過長
●ILP 完全無法求解
●不利於實驗重現與比較

因此本研究採用：

●固定 random seed 的隨機取樣
●保留 VM / PM 異質性
●保留真實 workload 分布特性
此作法亦符合多數 Google Trace 相關文獻的實驗慣例。

| 項目     | PlanetLab | Google Trace |
| ------ | --------- | ------------ |
| 資料性質   | 學術測試平台    | 真實資料中心       |
| VM 規模  | ~1000     | 數萬（取樣）       |
| PM 容量  | 無         | 有            |
| CPU    | 使用率       | 使用率          |
| Memory | 推估        | 真實使用量        |
| 實驗目的   | 方法驗證      | 規模與實務驗證      |


兩者互補，使實驗同時具備學術與實務價值。

實驗結果初步觀察（Google Trace）
根據目前結果：

●FFD / RLS-FFD
  ●使用較少 PM
  ●CPU utilization 較高
  ●能源消耗較低
●BFD
  ●使用較多 PM
  ●Placement 較分散
  ●執行時間顯著增加
