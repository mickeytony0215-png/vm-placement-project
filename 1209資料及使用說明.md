# Dataset & Real-World Data Usage

## 1. 資料來源 (Data Source)
- **名稱**: PlanetLab Workload Traces (20110303)
- **原始格式**: 來自全球分佈式伺服器的 CPU 使用率追蹤數據。
- **用途**: 作為真實世界負載的基礎，提供隨機且真實的 VM 初始分佈。

## 2. 資料處理流程 (Data Processing)

### 步驟 A: 載入與清洗 (Loading & Cleaning)
- 使用 `PlanetLabLoader` 讀取特定時間點 (Time Point = 144, 即中午 12:00) 的快照。
- **鍵值修正 (Key Fix)**: 將原始資料的 `cpu` / `ram` 欄位映射為演算法所需的 `cpu_demand` / `memory_demand`。

### 步驟 B: 壓力測試邏輯轉化 (Stress Test Transformation)
為了評估演算法在極端情況下的表現，我們不直接使用原始數值，而是將其轉化為「互補型」資源需求。這模擬了雲端環境中常見的 **Skewed Workload (偏斜負載)**。

**轉化邏輯 (`apply_stress_test_logic`)**:
1. **基礎數值 (Base Value)**: 對每個 VM 生成一個 10~20 之間的隨機基準值。
2. **類型 A (CPU 密集型)**:
   - CPU 需求 = `Base Value` (高)
   - RAM 需求 = `Base Value / 4` (低)
   - *模擬場景: 計算密集型任務，如加密運算、影像處理。*
3. **類型 B (記憶體密集型)**:
   - CPU 需求 = `Base Value / 4` (低)
   - RAM 需求 = `Base Value` (高)
   - *模擬場景: 緩存伺服器 (Redis/Memcached)、資料庫查詢。*

### 3. 設計目的
這種「互補型」設計強迫演算法必須進行 **完美配對 (Perfect Matching)**。
- 若將兩個 "類型 A" 放在一起 -> CPU 爆滿，RAM 閒置 (資源浪費)。
- 只有將 "類型 A" 與 "類型 B" 放在同一台機器，才能達成高資源利用率。
